# ТДМ — Roadmap проекта

> Проект: **ТДМ (Такси Для Марокко)** — WhatsApp-бот такси.
> Основная цель: спроектировать **FSM интерфейса WhatsApp-бота**, который выступает чистым
> представлением (UI-слоем) над **внешним FSM заказа**, не владея бизнес-логикой заказа.
>
> Дата составления: 2026-06-19. Статус: черновик, требует утверждения заказчиком.

---

## 0. Ключевые решения (приняты)

| Вопрос | Решение |
|---|---|
| Кодовая база нового бота | **Решаем после gap-анализа** обоих ботов (WATaxiBot vs MultiBot). |
| Владелец FSM заказа | **Считаем внешним и неизвестным.** Интерфейс бота проектируем через **абстрактный контракт**, не зависящий от реализации FSM заказа. |
| Первый артефакт | Этот roadmap. |

---

## 1. Архитектурный принцип (фундамент всего плана)

Разделяем **три независимых FSM**, которые сейчас в коде перемешаны:

```
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 1: FSM формы/диалога (per-channel)                      │
│  Сбор данных у клиента: from, to, места, время, опции…        │
│  Своя реализация на каждый канал (WhatsApp / web / mobile).   │
└───────────────────────────┬─────────────────────────────────┘
                            │ создаёт заказ (Order params)
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 3: FSM сопровождения заказа в боте  ← НАША ОСНОВНАЯ ЗАДАЧА │
│  Подписка на доменные события + отрисовка их клиенту в WA.     │
│  Никакой бизнес-логики заказа внутри — только представление.  │
└───────────────────────────┬─────────────────────────────────┘
                            │ слушает события / шлёт команды
                            ▼ (через абстрактный контракт)
┌─────────────────────────────────────────────────────────────┐
│  СЛОЙ 2: FSM ЗАКАЗА (внешний, «чужой»)                        │
│  CREATED → MATCHING → ASSIGNED → BOARDING → RIDE → FINISHED   │
│  Режимы DIRECT / VOTE / OFFER. Владеет внешняя система ТДМ.    │
└─────────────────────────────────────────────────────────────┘
```

**Инвариант:** бот никогда не вычисляет статус заказа сам — он только отражает события,
пришедшие из FSM заказа, и отправляет в него команды (создать/отменить/выбрать водителя).

## 2. Этапы работ

### Этап 1 — Доменный фундамент (документация)
**Цель:** единая непротиворечивая модель, на которую опирается весь проект.
*(grok1.txt прямо указал на расхождения между документами — устраняем их здесь.)*

- [ ] **1.1 Glossary** — единый словарь терминов (Order, Candidate, Offer, PoolVisibility, AssignedDriver, ActualCarrier, BoardingVerification, ExecutionOutcome…). Свести терминологию переписки + gpt3 + grok.
- [ ] **1.2 Доменная модель заказа** — структура Order (маршрут, тип поездки, класс, requirements HARD_FILTER/SOFT_SCORE, цены, таймстемпы, геолокация requested vs actual).
- [ ] **1.3 Модели исполнения** — DIRECT / VOTE / OFFER: участники, этапы, события, варианты завершения, таблица различий.
- [ ] **1.4 Carrier Determination** — детально (главный пробел по grok): стратегии ClientManualSelection / FirstArrived / AnyArrived / ExternalCarrier; когда CandidateSelected → DriverAssigned.

**Результат:** `docs/domain/` (glossary.md, order-model.md, execution-models.md).

---

### Этап 2 — Спецификация FSM заказа (внешний контракт)
**Цель:** описать «чужой» FSM как чёрный ящик, на события которого реагирует бот.

- [ ] **2.1 Состояния и переходы FSM заказа** — на базе gpt3.txt, выверенные по grok (добавить EN_ROUTE/HEADING_TO_PICKUP, Re-matching/Re-assignment).
- [ ] **2.2 Каталог доменных событий** — нормализованный список: `OrderCreated`, `DriverAssigned`, `DriverArrived`, `BoardingVerified`, `RideStarted`, `RideFinished`, `OrderCancelled`, `OrderExpired`, `CandidateAdded`, `OfferSubmitted` и т.д. Для каждого — payload.
- [ ] **2.3 Таймеры** — matching/candidate/offer/boarding/pickup; как влияют на переходы (cross-cutting слой, не состояния).
- [ ] **2.4 Команды боту → FSM заказа** — createOrder, cancelOrder, selectCandidate, selectOffer, confirmBoarding.

**Результат:** `docs/order-fsm/` (states.md, events.md, commands.md, timers.md) + Mermaid-диаграммы.

---

### Этап 3 — Контракт интеграции (бот ↔ FSM заказа)
**Цель:** абстрактный интерфейс, изолирующий бот от реализации внешнего FSM.
*(Транспорт неизвестен — проектируем под подмену: polling / webhook / WebSocket / очередь.)*

- [ ] **3.1 Порт интеграции** — интерфейс `OrderGateway`: `createOrder(params)`, `cancelOrder(id)`, `subscribe(orderId, onEvent)`, команды выбора.
- [ ] **3.2 Маппинг внешних статусов → нормализованные события** — таблица (как сейчас `b_state` → `order_status_*`), но вынесенная в конфиг/адаптер.
- [ ] **3.3 Стратегии доставки событий** — адаптеры: PollingAdapter (как сейчас), WebhookAdapter, WSAdapter — за одним интерфейсом.
- [ ] **3.4 Идемпотентность и доставка** — порядок событий, дубли, пропуски, восстановление после рестарта.

**Результат:** `docs/integration/order-gateway-contract.md`.

---

### Этап 4 — Спецификация FSM интерфейса бота (ОСНОВНАЯ ЗАДАЧА)
**Цель:** формальное описание двух ботовых FSM.

- [ ] **4.1 FSM формы (слой 1)** — состояния сбора данных заказа: from → to → people → when → options → carClass → confirm. Раздельно user-events и validation. Очищены от доменной логики заказа.
- [ ] **4.2 FSM сопровождения (слой 3)** — состояния после создания заказа: searching → assigned → driverArrived → inRide → finished → rate; ветки VOTE (выбор кандидата) / OFFER (выбор предложения) / отмены / таймауты. Входы — события из контракта (Этап 3).
- [ ] **4.3 Введение Guard в DSL** — `order.hasPhone == true`, `driverCount > 0` и т.п. (gpt1/gpt2 указали на отсутствие Guard).
- [ ] **4.4 Единая модель событий** — явное разделение UI Event / Domain Event / System Event (ключевая рекомендация gpt2).
- [ ] **4.5 Cross-flow переходы** — формализовать переходы между flows (сейчас `{{link:order#approved}}`).

**Результат:** `docs/bot-fsm/` (form-fsm.md, tracking-fsm.md, dsl-spec.md) + JSON-схемы-черновики.

---

### Этап 5 — Gap-анализ кода и выбор базы
**Цель:** на основе Этапов 1–4 решить — MultiBot, WATaxiBot или гибрид.
*(Методология — gpt4.txt: reverse-mapping → сопоставление → gap → план миграции.)*

- [ ] **5.1 Reverse-mapping WATaxiBot** — восстановить фактическую FSM из `switch/case` обработчиков.
- [ ] **5.2 Reverse-mapping MultiBot** — выверить JSON-схемы против целевой модели.
- [ ] **5.3 Сопоставление обоих с целевой моделью (Этап 4)** — что упрощено/пропущено/смешано.
- [ ] **5.4 Решение и обоснование** — выбрать базу, оценить объём миграции.
  - *Предварительная гипотеза:* MultiBot ближе к цели (декларативный FSM, Redis, мульти-канал, system-events уже есть), но не финальный вывод.

**Результат:** `docs/gap-analysis.md` + решение по базе.

---

### Этап 6 — План реализации / миграции
**Цель:** превратить спеки в инкрементальный план разработки.

- [ ] **6.1 Доработка движка** — Guard-DSL, единая событийная модель, OrderGateway-порт, персистентность менеджеров (Redis).
- [ ] **6.2 Перенос FSM формы и FSM сопровождения** в JSON-схемы.
- [ ] **6.3 Замена polling на абстрактный gateway** + адаптеры.
- [ ] **6.4 Тесты** — сценарии VOTE с external carrier, OFFER, reassignment, таймауты (по grok).
- [ ] **6.5 Поэтапное переключение** канала WhatsApp на новую FSM как источник истины.

**Результат:** `docs/implementation-plan.md` + бэклог задач.

## 3. Зависимости этапов

```
Этап 1 (домен)
   ├──> Этап 2 (FSM заказа) ──┐
   │                          ├──> Этап 4 (FSM бота) ──> Этап 5 (gap) ──> Этап 6 (реализация)
   └──> Этап 3 (контракт) ────┘
```

Этапы 2 и 3 можно вести параллельно после Этапа 1. Этап 5 требует готового Этапа 4.

## 4. Открытые вопросы к заказчику

1. **Транспорт событий заказа:** polling существующего API (как сейчас), webhook или WebSocket? От этого зависит Этап 3.
2. **Какие модели исполнения в MVP?** Только DIRECT или сразу VOTE/OFFER? (влияет на объём FSM сопровождения).
3. **Каналы:** только WhatsApp или сразу закладываем Telegram/web (MultiBot это умеет)?
4. **Судьба двух ботов:** новый проект с нуля на выбранной базе или эволюция существующего?
5. **Бэкенд заказа:** будет ли он переписываться под доменную модель ТДМ, или интегрируемся с текущим API (`b_state` и т.п.)?
