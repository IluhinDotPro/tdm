> **От:** Павел (аналитика/спецификации), по итогам сверки FSM-ядра с Иваном **2026-06-26**.
> **Что это:** короткий запрос на комментарий (RFC) **только по развитию самого движка**. Под Vote-заказ
> такси нам нужны декларативные **guard / effect** на переходах и **timer worker**. Хотим заложить их как
> механизмы ядра, **не сломав универсальность** движка (он вырос из доставки через постаматы — ты его
> носитель). Нужен твой ответ именно как автора ядра.

## Контекст: как мы понимаем движок сейчас (поправь, если не так)

Источник — дамп `vote_fsm` и описание Ивана. Если что-то из этого неверно — твоя правка авторитетна.

- Движок **табличный**: переходы в `fsm_states` / `fsm_actions` / `fsm_transitions`; исполнение — ХП
  `fsm_perform_action`; журнал переходов — `fsm_action_logs`.
- Состояние сущности хранится в её таблице (`orders.status`). Для taxi заведён `server_fsm_instances` —
  служебный слой исполнения, пока **без** worker/timers.
- **Отдельного guard-слоя в ХП нет** — условия сейчас проверяются в API/action layer **до** вызова
  `fsm_perform_action`.
- **Универсального timer subsystem нет** — таймеры состояния (expire, no-show) в рантайме не срабатывают.
- Side-effects перехода держатся в action layer / worker, **не внутри ХП**.

Цель — добавить guard/effect/timer как **универсальные механизмы движка**, а не taxi-only заплатки.
Там, где это противоречит архитектуре ядра, твоё «нет, так нельзя» для нас ценнее нашей гипотезы.

## Вопросы

**1. Расширяемость `fsm_transitions`.** Поддерживает ли архитектура ХП добавление в таблицу переходов
полей `guard_name`, `guard_params`, `effect_name`, `effect_params` — без слома существующего
(постаматного) движка? Это естественное расширение текущей модели или чужеродный для неё элемент?

**2. Где должен вызываться guard?** Какой вариант правильнее с точки зрения ядра:
- (a) **до** `fsm_perform_action` — в API/action layer (как сейчас де-факто);
- (b) **внутри** `fsm_perform_action`;
- (c) **после выбора перехода, но до фиксации состояния** (внутри транзакции перехода).

**3. Нужен ли registry уже сейчас?** Закладывать `GuardRegistry` / `EffectRegistry` (разрешение guard и
effect по имени) сразу — или пока оставить только Action Layer (имена разрешаются в приложении), а
registry ввести позже?

**4. Как организовать Timer Worker?** Worker по `server_fsm_instances.next_timer_at` должен:
- (a) **напрямую менять состояние** (`UPDATE` статуса);
- (b) **вызывать `fsm_perform_action(timeout)`** — таймаут как обычное действие;
- (c) использовать **тот же механизм команд**, что и пользовательские действия (timeout = синтетическое
  событие через общий вход, с guard и журналом).

Что лучше ложится на движок и не делает таймер «обходным путём» мимо журнала/guard?

**5. Поля сейчас или позже?** Имеет ли смысл добавить `guard_name` / `effect_name` в `fsm_transitions`
**уже сейчас** (пустыми / NULL), чтобы не делать миграцию схемы позже, когда механизм заработает?

**6. Ограничения текущей архитектуры.** Есть ли в текущей ХП-архитектуре ограничения, которые могут
помешать такому развитию (guard / effect / timer / registry)? Что ты как автор ядра видишь как риск —
и для устойчивости движка, и для его универсальности?
